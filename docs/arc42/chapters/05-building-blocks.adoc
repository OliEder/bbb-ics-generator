== Bausteinsicht

=== Whitebox-Gesamtsystem

Das System besteht aus fünf aktiven Modulen. Die Module sind lose gekoppelt und kommunizieren ausschließlich über Funktionsaufrufe (CommonJS `require`).

[source,mermaid]
----
graph LR
    CU["cronUpdate.js\nOrchestrator"] --> AC["apiClient.js\nAPI-Facade"]
    CU --> IG["icsGenerator.js\nRFC 5545 Renderer"]
    CU --> ST["storage.js\nPersistenz"]
    CU --> GH["generateHTML.js\nPortal-Generator"]
    GH --> ST
    SV["server.js\nDev-Server"] -.->|nur lokal| ST
----

=== Modul: cronUpdate.js (Orchestrator)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Einstiegspunkt des Update-Prozesses. Orchestriert den kompletten Daten-Refresh: Teams ermitteln, Spielpläne abrufen, ICS generieren, Dateien speichern, Metadaten schreiben.

| *Schnittstellen (eingehend)*
| Keine. Wird direkt via `node src/cronUpdate.js` gestartet (durch GitHub Actions oder manuell).

| *Schnittstellen (ausgehend)*
| `apiClient.js`: `fetchTeamMatches(teamId)`, `fetchMatchInfo(matchId)`, `fetchClubTeams(clubId)` +
`icsGenerator.js`: `generateICS(matches, details, teamId, kind, teamName)` +
`storage.js`: `saveICS(teamId, type, data)`, `saveTeamsCache(teams)`, `loadTeamsCache()` +
`config.json`: `clubId`

| *Wichtige Logik*
| Zweistufige Strategie für Team-Ermittlung: (1) Cache-HIT → sofortige Nutzung, (2) Cache-MISS → API-Aufruf + Cache-Befüllung, (3) API-Fehler + abgelaufener Cache → Fallback auf alten Cache. Pro Team werden drei ICS-Varianten erzeugt: `all`, `home`, `away`. Pro Team werden zusätzlich `spotlightMatches` (0–2 Einträge) berechnet via `computeSpotlight`: letztes Ergebnis + nächstes Spiel, oder letzte 2 Ergebnisse, oder erste 2 zukünftige.

| *Fehlerbehandlung*
| Fehler pro Team werden geloggt, aber der Gesamtprozess wird für die verbleibenden Teams fortgesetzt (Fail-Graceful per Team-Isolation in try/catch).

| *Abhängigkeiten*
| `./apiClient`, `./icsGenerator`, `./storage`, `../config.json`, `fs`, `path`
|===

=== Modul: apiClient.js (API-Facade)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Kapselt alle HTTP-Aufrufe an die Basketball-Bund REST-API. Abstrahiert die API-Struktur und gibt normalisierte Daten zurück. Fehlerbehandlung auf Transport-Ebene.

| *Exportierte Funktionen*
a|
* `fetchTeamMatches(teamId)` → `Array<Match>` (leer bei Fehler)
* `fetchMatchInfo(matchId)` → `MatchInfo \| null` (null bei Fehler)
* `fetchClubTeams(clubId)` → `Array<Team> \| null` (null bei Fehler, leer wenn keine Matches)

| *Schnittstellen (ausgehend)*
| Basketball-Bund REST-API via `axios` HTTP-Client. Basis-URL: `https://www.basketball-bund.net/rest`.

| *Wichtige Logik*
| `fetchClubTeams` dedupliziert Teams per `teamPermanentId` (Set-basiertes Seen-Tracking). Nur Teams, deren `clubId` mit der übergebenen `clubId` übereinstimmt, werden zurückgegeben.

| *Fehlerbehandlung*
| Alle Funktionen fangen HTTP-Fehler und Netzwerkfehler via try/catch. Bei Fehler: `fetchTeamMatches` → `[]`, `fetchMatchInfo` → `null`, `fetchClubTeams` → `null`.

| *Abhängigkeiten*
| `axios`
|===

=== Modul: icsGenerator.js (RFC 5545 Renderer)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Transformiert rohe Spielplan-Daten in valide ICS-Kalender-Dateien nach RFC 5545. Handhabt Zeitberechnung, Beschreibungsformatierung, ICS-Escaping und VCALENDAR-Header-Injection.

| *Exportierte Funktionen*
a|
* `generateICS(matches, details, teamId, type, teamName)` → `Promise<string \| null>`

| *Interne Funktionen*
a|
* `buildEvent(match, matchInfo, teamId, calendarType)` → `Promise<EventObject>`
* `icsEscape(text)` → escaping nach RFC 5545 (Backslash, Semikolon, Komma, Newline)
* `escapeHtml(str)` → HTML-Escaping für X-ALT-DESC
* `createHtmlDescription(lines, feld)` → HTML-Snippet für X-ALT-DESC
* `dateToArr(d)` → Date-Objekt zu `[year, month, day, hour, minute]` Array
* `formatKickoff(dateStr, timeStr)` → Deutsche Datumsformatierung für Beschreibungstext

| *Zeitberechnung*
a|
* `dtstart` = Kickoff − 60 Minuten (Anreisepuffer)
* `dtend` = Kickoff + 150 Minuten (2,5 Stunden Spielzeit inkl. Puffer)
* Alarm: 30 Minuten vor `dtstart` bei Heimspielen, 60 Minuten bei Auswärtsspielen

| *Wichtige Logik*
| Die `ics`-Bibliothek erzeugt RFC-konforme VEVENT-Blöcke. Der Generator führt danach eine zeilenbasierte Post-Processing-Pipeline aus: VCALENDAR-Header-Ersetzung mit X-WR-*-Properties, X-ALT-DESC-Injektion nach DESCRIPTION unter Berücksichtigung von Line-Folding, Entfernung automatisch generierter Duplikat-Header.

| *ICS-Escaping*
| `icsEscape` kodiert die vier RFC-5545-Sonderzeichen: `\` → `\\`, `;` → `\;`, `,` → `\,`, `\n` → `\n` (literal), `\r` wird entfernt. HTML-Beschreibungen werden zusätzlich von echten Newlines bereinigt.

| *Abhängigkeiten*
| `ics` (npm-Paket v3)
|===

=== Modul: storage.js (Persistenz)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Kapselt alle Lese- und Schreibzugriffe auf das Dateisystem. Stellt sicher, dass keine Path-Traversal-Angriffe möglich sind. Verwaltet den Team-Cache mit TTL.

| *Exportierte Funktionen*
a|
* `saveICS(teamId, type, data)` → `string` (Dateipfad)
* `readICS(teamId, type)` → `string \| null`
* `saveTeamsCache(teams)` → schreibt `teams-cache.json`
* `loadTeamsCache()` → `{ teams: Array \| null, stale: boolean }`

| *Sicherheitsmechanismen*
a|
* *Whitelist-Validierung für `type`*: Nur `'all'`, `'home'`, `'away'` erlaubt (via `Set`-Lookup).
* *Regex-Validierung für `teamId`*: Nur reine Zahlenketten erlaubt (`/^\d+$/`).
* Beide Prüfungen verhindern Path-Traversal (z. B. `../../etc/passwd`).

| *Cache-Logik*
| `teams-cache.json` enthält `{ cachedAt: ISO-8601-String, teams: [...] }`. TTL = 30 Tage. `loadTeamsCache` gibt `stale: true` zurück wenn `Date.now() - cachedAt >= 30 * 24 * 60 * 60 * 1000`. Bei Parse-Fehler wird `{ teams: null, stale: true }` zurückgegeben.

| *Ausgabeverzeichnis*
| `generated/` (relativ zum Projektroot). Wird bei Start automatisch angelegt wenn nicht vorhanden.

| *Abhängigkeiten*
| `fs`, `path` (Node.js Stdlib)
|===

=== Modul: generateHTML.js (Portal-Generator)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Erzeugt die komplette statische Website aus `metadata.json`: Startseite (`index.html`) mit adaptiven Team-Teasern und Navigationsmenü sowie individuelle Team-Seiten (`teams/{teamId}.html`) mit Next-Game-Teaser, Ligatabellen, Turnierbäumen, Spielplan-Tabs und Kalender-Abo-Links.

| *Eingang*
| `generated/metadata.json` mit vollständigen Team-Metadaten: Spielplan (mit `isNext`, `ownShort`, `opponentShort`, `venueName`, `venueAddress`), Ligatabellen (`competitions[].table`), Turnierbäume (`competitions[].bracket`), Geschlecht, Altersgruppe, Logo-URL, `spotlightMatches` (0–2 Spotlight-Spiele pro Team).

| *Ausgang*
| `generated/index.html` (Startseite), `generated/teams/{teamId}.html` (eine Datei pro Team), `generated/datenschutz.html` und `generated/barrierefreiheit.html` (immer), sowie `generated/impressum.html` (nur wenn `config.legal` mindestens ein nicht-leeres Feld hat). Alle HTML-Dateien sind vollständig eigenständig (inline CSS + JS, keine externen Assets außer Leaflet CDN, Font Awesome CDN und BBB-Logo-URL).

| *Wichtige interne Funktionen*
a|
* `buildTeaserCard(team)` — adaptiver Teaser mit bis zu 3 vergangenen Ergebnissen + bis zu 4−n zukünftigen Terminen
* `buildNextGameTeaser(team)` — detaillierter Next-Game-Block mit Leaflet-Karte (clientseitig per Nominatim geocodiert)
* `buildSpotlightBlock(teams, cupColor)` — "Nächste Spiele"-Block auf der Startseite mit Tabs Alle/Heim/Auswärts, chronologisch sortiert über alle Teams
* `buildStandingsTable(comp)` — zwei Tabs: offizielle Tabelle + Games-Behind-Ranking
* `buildBracket(comp, ownTeamName)` — Turnierbaum mit Highlight des eigenen Teams und TBD-Platzhaltern für zukünftige Runden
* `buildTeamPage(team, allTeams, theme)` — vollständige Team-Seite
* `buildIndexPage(teams, theme)` — Startseite
* `teamLabel(teamName, ageGroup, gender)` — einheitliche Teambezeichnung mit Geschlechtssymbol
* `sortTeams(teams)` — Senioren zuerst, dann aufsteigend nach Altersgruppe
* `buildFooter(legal, relativePath)` — rendert `<footer role="contentinfo">` mit Quellenlink zu basketball-bund.net, optionalem Impressum-Link (nur wenn `legal` mindestens ein nicht-leeres Feld hat), Datenschutz-Link und Barrierefreiheit-Link. `relativePath` ist `'./'` für Index- und Legal-Seiten, `'../'` für Team-Seiten. Wird in `buildIndexPage`, `buildTeamPage` und allen Legal-Page-Buildern verwendet.
* `buildImpressum(legal, allTeams, theme)` — erzeugt `generated/impressum.html` nur wenn `config.legal` mindestens ein nicht-leeres Feld hat. Zeigt Betreiber, Adresse, Telefon (optional), E-Mail und Verantwortlichen (optional) aus der Konfiguration sowie einen Haftungshinweis zu aggregierten öffentlichen Daten.
* `buildDatenschutz(legal, allTeams, theme)` — erzeugt `generated/datenschutz.html` (wird immer generiert). Nennt Drittanbieter: cdnjs.cloudflare.com (Font Awesome), basketball-bund.net (Spielplandaten + Logos). Betreiber und E-Mail werden aus `config.legal` bezogen.
* `buildBarrierefreiheit(legal, allTeams, theme)` — erzeugt `generated/barrierefreiheit.html` (wird immer generiert). Status: "Teilweise konform mit WCAG 2.1 Stufe AA". E-Mail-Kontakt aus der Konfiguration.
* `genHTML(theme, legal)` — Einstiegsfunktion; `legal`-Parameter wurde hinzugefügt (vorher `genHTML(theme)`).

| *Abhängigkeiten*
| `fs`, `path` (Node.js Stdlib); Leaflet 1.9.4 (CDN, nur in Team-Seiten); Font Awesome (cdnjs.cloudflare.com, CDN)
|===

=== Modul: server.js (Lokaler Entwicklungsserver)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Stellt die generierten Dateien lokal unter `http://localhost:3000` bereit, damit Entwickler die Ausgabe ohne GitHub Pages testen können.

| *Verwendung*
| Ausschließlich für lokale Entwicklung. Wird in GitHub Actions nicht aufgerufen.

| *Abhängigkeiten*
| `express`
