== Einführung und Ziele

=== Aufgabenstellung

Der *BBB ICS Generator* ist ein automatisierter Kalender-Feed-Dienst für die *Fibalon Baskets Neumarkt*.
Er ruft Spieltermine für alle Teams des Vereins über die öffentliche REST-API des Deutschen Basketball-Bundes (DBB) ab und stellt diese als abonnierbare ICS-Feeds bereit.

Die Feeds können von Fans, Spielern und Betreuern direkt in ihre Kalenderanwendungen (iOS Calendar, Google Calendar, Outlook, Thunderbird u. a.) abonniert werden.
Nach einmaliger Einrichtung erhalten Abonnenten automatisch alle neuen und geänderten Spieltermine, ohne manuelles Eingreifen.

Die Kernfunktionalität umfasst:

* Automatischer Abruf aller aktiven Teams des Vereins über die `clubId`
* Abruf der Spielplandaten (Gegner, Datum, Uhrzeit, Liga, Spielfeld) pro Team
* Erzeugung valider ICS-Dateien nach RFC 5545 in drei Varianten: alle Spiele, nur Heimspiele, nur Auswärtsspiele
* Bereitstellung als statische Dateien über GitHub Pages mit webcal://, HTTPS- und Google-Calendar-Links
* Automatische Aktualisierung alle sechs Stunden über GitHub Actions

== Randbedingungen

=== Technische Randbedingungen

[cols="2,4,3", options="header"]
|===
| Randbedingung | Beschreibung | Konsequenz

| *Node.js 18*
| Die GitHub Actions-Pipeline verwendet explizit `node-version: '18'`. Alle verwendeten npm-Pakete müssen mit Node.js 18 kompatibel sein.
| Kein Einsatz von Node.js-21+-only-Features; Abhängigkeiten müssen LTS-kompatibel sein.

| *RFC 5545 (iCalendar)*
| Die erzeugten Dateien müssen dem iCalendar-Standard RFC 5545 entsprechen. Pflichtfelder (`DTSTART`, `DTEND`, `UID`, `DTSTAMP`) müssen vorhanden sein. Sonderzeichen müssen nach ICS-Escape-Regeln kodiert werden. Zeilen dürfen maximal 75 Oktetts lang sein (Line-Folding).
| Das Bibliothek-Paket `ics` (v3) übernimmt das grundlegende Rendering; manuelle Nachbearbeitung ist für X-Properties und VCALENDAR-Header erforderlich.

| *Statisches Hosting (GitHub Pages)*
| Alle öffentlich zugänglichen Dateien müssen statisch sein. Kein serverseitiges Rendering, keine Datenbanken, keine laufenden Prozesse auf GitHub Pages.
| Alle ICS-Dateien und die HTML-Übersichtsseite werden im `generated/`-Verzeichnis vorgebaut und als Artefakt deployed.

| *Keine Infrastrukturkosten*
| Das gesamte System muss im kostenlosen Tier von GitHub betrieben werden können (GitHub Actions Free Tier: 2.000 Minuten/Monat für öffentliche Repos; GitHub Pages: kostenlos für öffentliche Repos).
| Alle 6 Stunden läuft ein Build von ca. 1–2 Minuten = ca. 240 Builds/Monat * 2 min = 480 min/Monat — gut im Rahmen.

| *Öffentliche API ohne Authentifizierung*
| Die Basketball-Bund REST-API unter `https://www.basketball-bund.net/rest` ist öffentlich zugänglich und erfordert keine API-Keys oder OAuth.
| Kein Secret-Management erforderlich. Kein Rate-Limiting durch Authentifizierungsebene — Risiko von IP-basiertem Rate-Limiting bleibt.

| *Abhängigkeit von npm-Paketen*
| Verwendete Pakete: `axios` (HTTP-Client), `ics` (ICS-Rendering), `express` (lokaler Dev-Server), `node-cron` (lokaler Cron-Trigger).
| Sicherheitsrelevante Updates müssen durch Ausführen von `npm install` im CI-Lauf aufgenommen werden. Kein Lockfile-Enforcement in der Pipeline.

| *Kein persistenter Speicher in GitHub Actions*
| GitHub Actions-Runner sind ephemer. Der `teams-cache.json` wird bei jedem Lauf neu erzeugt, da das `generated/`-Verzeichnis nicht zwischen Runs persistiert wird.
| Der 30-Tage-TTL-Cache für Teams hat effektiv keine Wirkung in der GitHub-Actions-Umgebung; er ist nur für lokale Entwicklung relevant. Bei jedem CI-Lauf wird die Club-API erneut aufgerufen.

== Lösungsstrategie

=== Überblick der grundlegenden Entscheidungen

Die Architektur des BBB ICS Generators basiert auf vier zentralen strategischen Entscheidungen, die gemeinsam ein einfaches, robustes und wartungsarmes System ergeben.

=== Warum ICS statt CalDAV?

ICS (RFC 5545) ist ein Pull-basiertes, dateibasiertes Format. Der Client ruft die Datei periodisch ab und synchronisiert lokal. CalDAV ist ein serverbasiertes Protokoll, das eine aktive Serverinstanz mit persistentem Speicher, Authentifizierung und Synchronisationslogik erfordert.

*Entscheidung:* ICS-Dateien als statische Ressourcen.

*Begründung:*

* Statische Dateien können auf GitHub Pages gehostet werden — kostenlos, wartungsfrei, hochverfügbar.
* Alle gängigen Kalenderanwendungen (iOS, Google, Outlook, Thunderbird) unterstützen ICS-Subscriptions nativ.
* Kein Server, keine Datenbank, keine Laufzeitumgebung erforderlich.
* Ausfälle des Generators beeinträchtigen bestehende Subscriptions nicht — der Client zeigt schlimmstenfalls veraltete Daten, aber keine Fehler.

=== Warum GitHub Pages als Hosting?

GitHub Pages ist das natürliche Hosting-Pendant zu GitHub Actions für statische Inhalte.

*Entscheidung:* GitHub Pages als alleinige Hosting-Plattform.

*Begründung:*

* Kostenlos für öffentliche Repositories ohne Bandbreitenbeschränkung für typische Kalender-Payloads.
* Nahtlose Integration mit GitHub Actions über `actions/upload-pages-artifact` und `actions/deploy-pages`.
* HTTPS ist out-of-the-box vorhanden — Voraussetzung für `webcal://`-Subscriptions auf modernen iOS-Versionen.
* Keine separate Infrastruktur, kein separates Deployment-System.
* Verfügbarkeit von GitHub Pages liegt typischerweise bei über 99,9 % — ausreichend für einen Kalender-Feed.

=== Warum ein 30-Tage-Cache für Teams?

Die Team-Metadaten (welche Teams gehören zum Verein?) ändern sich selten — typischerweise einmal pro Saison, wenn neue Altersklassen-Teams dazukommen oder wegfallen.

*Entscheidung:* `teams-cache.json` mit 30-Tage-TTL.

*Begründung:*

* Jeder API-Aufruf an `/rest/club/id/{clubId}/actualmatches` kostet Latenz und belastet die externe API.
* Bei 4 Läufen pro Tag und ~8 Teams wären das ohne Cache zusätzliche ~120 API-Aufrufe pro Monat allein für Teamdaten.
* 30 Tage deckt den gesamten Zeitraum ab, in dem sich Teamzusammensetzungen nicht ändern.
* Bei Cache-Ablauf wird die API erneut aufgerufen; bei API-Fehler wird der abgelaufene Cache als Fallback verwendet.

*Hinweis:* In der GitHub-Actions-Umgebung ist das `generated/`-Verzeichnis nicht persistent zwischen Runs. Der Cache wirkt dort de facto nicht — er ist primär für lokale Entwicklungsumgebungen relevant. Dieser Sachverhalt ist als technische Schuld dokumentiert (siehe Kapitel 11).

=== Warum eine einzige clubId als Konfiguration?

Das System ist als Single-Verein-Lösung konzipiert. Die gesamte Konfiguration besteht aus einer einzigen Datei mit einer einzigen Variablen.

*Entscheidung:* `config.json` mit nur `{ "clubId": "4468" }`.

*Begründung:*

* Einfachheit hat Priorität. Der Betreiber ist Mitglied des Vereins und hat ein klar definiertes Einsatzszenario.
* Alle Teams des Vereins werden automatisch über die Club-API ermittelt — keine manuelle Pflege einer Teamliste.
* Ein neues Team wird automatisch beim nächsten Cache-Ablauf erkannt.
* Für andere Vereine kann eine separate Instanz des Systems mit einer anderen `clubId` betrieben werden.

=== Zusammenfassung der strategischen Entscheidungen

[cols="3,3,3", options="header"]
|===
| Strategie | Alternative | Ausschlaggebendes Kriterium

| ICS (RFC 5545) als Ausgabeformat
| CalDAV, proprietäre APIs
| Universelle Kompatibilität, kein Server nötig

| GitHub Pages als Hosting
| Eigener Server, Netlify, Vercel, S3
| Keine Kosten, nahtlose GitHub-Integration

| 30-Tage-Cache für Teamdaten
| Kein Cache (immer API-Aufruf), kein TTL
| API-Effizienz, Ausfalltoleranz

| Eine clubId als einzige Konfiguration
| Multi-Club-Konfiguration, Datenbank
| Einfachheit, Self-Service durch Club-API
|===

=== Modul: cronUpdate.js (Orchestrator)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Einstiegspunkt des Update-Prozesses. Orchestriert den kompletten Daten-Refresh: Teams ermitteln, Spielpläne abrufen, ICS generieren, Dateien speichern, Metadaten schreiben.

| *Schnittstellen (eingehend)*
| Keine. Wird direkt via `node src/cronUpdate.js` gestartet (durch GitHub Actions oder manuell).

| *Schnittstellen (ausgehend)*
| `apiClient.js`: `fetchTeamMatches(teamId)`, `fetchMatchInfo(matchId)`, `fetchClubTeams(clubId)` +
`icsGenerator.js`: `generateICS(matches, details, teamId, kind, teamName)` +
`storage.js`: `saveICS(teamId, type, data)`, `saveTeamsCache(teams)`, `loadTeamsCache()` +
`config.json`: `clubId`

| *Wichtige Logik*
| Zweistufige Strategie für Team-Ermittlung: (1) Cache-HIT → sofortige Nutzung, (2) Cache-MISS → API-Aufruf + Cache-Befüllung, (3) API-Fehler + abgelaufener Cache → Fallback auf alten Cache. Pro Team werden drei ICS-Varianten erzeugt: `all`, `home`, `away`.

| *Fehlerbehandlung*
| Fehler pro Team werden geloggt, aber der Gesamtprozess wird für die verbleibenden Teams fortgesetzt (Fail-Graceful per Team-Isolation in try/catch).

| *Abhängigkeiten*
| `./apiClient`, `./icsGenerator`, `./storage`, `../config.json`, `fs`, `path`
|===

=== Modul: apiClient.js (API-Facade)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Kapselt alle HTTP-Aufrufe an die Basketball-Bund REST-API. Abstrahiert die API-Struktur und gibt normalisierte Daten zurück. Fehlerbehandlung auf Transport-Ebene.

| *Exportierte Funktionen*
a|
* `fetchTeamMatches(teamId)` → `Array<Match>` (leer bei Fehler)
* `fetchMatchInfo(matchId)` → `MatchInfo \| null` (null bei Fehler)
* `fetchClubTeams(clubId)` → `Array<Team> \| null` (null bei Fehler, leer wenn keine Matches)

| *Schnittstellen (ausgehend)*
| Basketball-Bund REST-API via `axios` HTTP-Client. Basis-URL: `https://www.basketball-bund.net/rest`.

| *Wichtige Logik*
| `fetchClubTeams` dedupliziert Teams per `teamPermanentId` (Set-basiertes Seen-Tracking). Nur Teams, deren `clubId` mit der übergebenen `clubId` übereinstimmt, werden zurückgegeben.

| *Fehlerbehandlung*
| Alle Funktionen fangen HTTP-Fehler und Netzwerkfehler via try/catch. Bei Fehler: `fetchTeamMatches` → `[]`, `fetchMatchInfo` → `null`, `fetchClubTeams` → `null`.

| *Abhängigkeiten*
| `axios`
|===

=== Modul: icsGenerator.js (RFC 5545 Renderer)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Transformiert rohe Spielplan-Daten in valide ICS-Kalender-Dateien nach RFC 5545. Handhabt Zeitberechnung, Beschreibungsformatierung, ICS-Escaping und VCALENDAR-Header-Injection.

| *Exportierte Funktionen*
a|
* `generateICS(matches, details, teamId, type, teamName)` → `Promise<string \| null>`

| *Interne Funktionen*
a|
* `buildEvent(match, matchInfo, teamId, calendarType)` → `Promise<EventObject>`
* `icsEscape(text)` → escaping nach RFC 5545 (Backslash, Semikolon, Komma, Newline)
* `escapeHtml(str)` → HTML-Escaping für X-ALT-DESC
* `createHtmlDescription(lines, feld)` → HTML-Snippet für X-ALT-DESC
* `dateToArr(d)` → Date-Objekt zu `[year, month, day, hour, minute]` Array
* `formatKickoff(dateStr, timeStr)` → Deutsche Datumsformatierung für Beschreibungstext

| *Zeitberechnung*
a|
* `dtstart` = Kickoff − 60 Minuten (Anreisepuffer)
* `dtend` = Kickoff + 150 Minuten (2,5 Stunden Spielzeit inkl. Puffer)
* Alarm: 30 Minuten vor `dtstart` bei Heimspielen, 60 Minuten bei Auswärtsspielen

| *Wichtige Logik*
| Die `ics`-Bibliothek erzeugt RFC-konforme VEVENT-Blöcke. Der Generator führt danach eine zeilenbasierte Post-Processing-Pipeline aus: VCALENDAR-Header-Ersetzung mit X-WR-*-Properties, X-ALT-DESC-Injektion nach DESCRIPTION unter Berücksichtigung von Line-Folding, Entfernung automatisch generierter Duplikat-Header.

| *ICS-Escaping*
| `icsEscape` kodiert die vier RFC-5545-Sonderzeichen: `\` → `\\`, `;` → `\;`, `,` → `\,`, `\n` → `\n` (literal), `\r` wird entfernt. HTML-Beschreibungen werden zusätzlich von echten Newlines bereinigt.

| *Abhängigkeiten*
| `ics` (npm-Paket v3)
|===

=== Modul: storage.js (Persistenz)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Kapselt alle Lese- und Schreibzugriffe auf das Dateisystem. Stellt sicher, dass keine Path-Traversal-Angriffe möglich sind. Verwaltet den Team-Cache mit TTL.

| *Exportierte Funktionen*
a|
* `saveICS(teamId, type, data)` → `string` (Dateipfad)
* `readICS(teamId, type)` → `string \| null`
* `saveTeamsCache(teams)` → schreibt `teams-cache.json`
* `loadTeamsCache()` → `{ teams: Array \| null, stale: boolean }`

| *Sicherheitsmechanismen*
a|
* *Whitelist-Validierung für `type`*: Nur `'all'`, `'home'`, `'away'` erlaubt (via `Set`-Lookup).
* *Regex-Validierung für `teamId`*: Nur reine Zahlenketten erlaubt (`/^\d+$/`).
* Beide Prüfungen verhindern Path-Traversal (z. B. `../../etc/passwd`).

| *Cache-Logik*
| `teams-cache.json` enthält `{ cachedAt: ISO-8601-String, teams: [...] }`. TTL = 30 Tage. `loadTeamsCache` gibt `stale: true` zurück wenn `Date.now() - cachedAt >= 30 * 24 * 60 * 60 * 1000`. Bei Parse-Fehler wird `{ teams: null, stale: true }` zurückgegeben.

| *Ausgabeverzeichnis*
| `generated/` (relativ zum Projektroot). Wird bei Start automatisch angelegt wenn nicht vorhanden.

| *Abhängigkeiten*
| `fs`, `path` (Node.js Stdlib)
|===

=== Modul: generateHTML.js (HTML-Generator)

[cols="2,5", options="header"]
|===
| Aspekt | Beschreibung

| *Verantwortlichkeit*
| Erzeugt die statische HTML-Übersichtsseite `generated/index.html` mit Subscription-Links für alle Teams.

| *Eingang*
| `generated/metadata.json` (erzeugt von `cronUpdate.js`) mit Teamname, Altersgruppe, Spielanzahlen und Timestamp.

| *Ausgang*

== Laufzeitsicht

=== Szenario 1: Update-Zyklus mit Cache-HIT (Normalfall)

Dieser Ablauf tritt ein, wenn der Team-Cache jünger als 30 Tage ist. Dies ist der reguläre Ablauf für die meisten der täglich vier Ausführungen.

[source,mermaid]
----
sequenceDiagram
    participant GHA as GitHub Actions
    participant CU as cronUpdate.js
    participant ST as storage.js
    participant AC as apiClient.js
    participant IG as icsGenerator.js
    GHA->>CU: npm run update
    CU->>ST: loadTeamsCache()
    ST-->>CU: {teams, stale:false}
    loop fuer jedes Team
        CU->>AC: fetchTeamMatches(teamId)
        AC-->>CU: Array<Match>
        loop fuer jedes Match
            CU->>AC: fetchMatchInfo(matchId)
            AC-->>CU: MatchInfo
        end
        CU->>IG: generateICS(matches, details, teamId, type)
        IG-->>CU: ICS-String
        CU->>ST: saveICS(teamId, type, ics)
    end
    CU->>CU: schreibt metadata.json
    CU->>CU: genHTML(theme)
----

==== Ablauf als nummerierte Schritte

. GitHub Actions startet den Runner und führt `node src/cronUpdate.js` aus.
. `cronUpdate.js` ruft `loadTeamsCache()` auf.
. `storage.js` liest `generated/teams-cache.json`. Das `cachedAt`-Datum ist weniger als 30 Tage alt.
. `loadTeamsCache()` gibt `{ teams: [...], stale: false }` zurück.
. `cronUpdate.js` nutzt die gecachten Teams direkt (kein API-Aufruf für Club-Teams).
. Für jedes Team: `fetchTeamMatches(teamId)` liefert alle Spiele des Teams als Array.
. Für jedes Match: `fetchMatchInfo(matchId)` liefert Detailinfos (Hallenname, Adresse, vollständige Teamnamen).
. Die Matches werden in drei Gruppen aufgeteilt: alle (`all`), nur Heimspiele (`home`), nur Auswärtsspiele (`away`).
. Für jede Gruppe: `generateICS(...)` erzeugt einen vollständigen ICS-String.
. In `buildEvent()` werden Zeiten berechnet: dtstart = Kickoff − 60 min, dtend = Kickoff + 150 min.
. Der Alarm wird gesetzt: 30 min vor dtstart bei Heimspiel, 60 min bei Auswärtsspiel.
. `createEvents()` der ics-Bibliothek erzeugt RFC-5545-konforme VEVENT-Blöcke.
. Post-Processing: VCALENDAR-Header wird mit X-WR-CALNAME, X-WR-TIMEZONE etc. angereichert.
. `saveICS(teamId, type, ics)` schreibt die Datei nach `generated/{teamId}_{type}.ics`.
. Nach allen Teams: `metadata.json` wird je Team mit Update-Zeitstempel, Spielzahlen, `logoUrl` und einem `matches`-Array geschrieben. Das `matches`-Array enthält alle Spiele der aktuellen Saison (seasonId 2025), gefiltert aus den Roh-API-Daten, aufsteigend nach Datum sortiert. Jeder Eintrag enthält: `date`, `time`, `opponent`, `isHome`, `result` (null = noch nicht gespielt), `competition` (Liga-Name), `isNext` (erstes Spiel ohne Ergebnis). Die `logoUrl` wird direkt konstruiert: `https://www.basketball-bund.net/media/team/{teamPermanentId}/logo` — kein zusätzlicher API-Aufruf.
. GitHub Actions führt danach `npm run generate-html` aus, das `index.html` aus `metadata.json` erzeugt (Tab-UI mit Logos, ARIA-Tab-Pattern, eingebettetem Keyboard-Nav-JS). Jeder Tab-Panel zeigt oben den gefilterten Spielplan (Alle/Heim/Auswärts) mit H/A-Badges und Wettbewerbs-Label, darunter die Kalender-Abo-Buttons.
. Das `generated/`-Verzeichnis wird als GitHub Pages Artefakt hochgeladen und deployed.

=== Szenario 2: Update-Zyklus mit Cache-MISS (erster Start oder Cache abgelaufen)

Dieser Ablauf tritt beim ersten Start oder wenn der Cache älter als 30 Tage ist.

==== Ablauf als nummerierte Schritte

. GitHub Actions startet den Runner und führt `node src/cronUpdate.js` aus.
. `cronUpdate.js` ruft `loadTeamsCache()` auf.
. `storage.js` findet entweder keine `teams-cache.json` oder das `cachedAt`-Datum ist älter als 30 Tage.
. `loadTeamsCache()` gibt `{ teams: null, stale: false }` (kein Cache) oder `{ teams: [...], stale: true }` (abgelaufen) zurück.
. `cronUpdate.js` erkennt den Cache-MISS (`!cached || stale`) und ruft `fetchClubTeams(clubId)` auf.
. `apiClient.js` sendet GET-Anfrage an `https://www.basketball-bund.net/rest/club/id/4468/actualmatches?justHome=false&rangeDays=150`.
. Die API antwortet mit einer Liste aller Spiele des Vereins in den nächsten 150 Tagen.
. `fetchClubTeams` extrahiert unique Teams (per `teamPermanentId`): iteriert über `homeTeam` und `guestTeam` jedes Spiels, filtert nach `clubId`, dedupliziert via Set.
. Das Ergebnis ist ein Array von `{ id, name, ageGroup }` Objekten.
. `saveTeamsCache(teams)` schreibt `{ cachedAt: <jetzt>, teams: [...] }` nach `generated/teams-cache.json`.
. Ab hier identisch mit Szenario 1, Schritt 6 ff.

==== Fehlerfall: Club-API nicht erreichbar

Wenn Schritt 6 fehlschlägt (HTTP-Fehler oder Netzwerkfehler):

. `fetchClubTeams` gibt `null` zurück.
. `cronUpdate.js` prüft: `fresh === null` → prüft ob `cached` (abgelaufener Cache) vorhanden.
. Falls abgelaufener Cache vorhanden: Warnung wird geloggt, abgelaufene Teams werden als Fallback verwendet.
. Falls kein Cache vorhanden: Fehlermeldung wird geloggt, `[]` wird zurückgegeben — kein ICS-Update in diesem Lauf.

== Querschnittliche Konzepte

=== Fehlerbehandlung (Fail-Graceful-Prinzip)

Das System folgt dem Prinzip der ausfalltoleranten Weiterführung: Ein Fehler bei einem einzelnen Team oder einer einzelnen API-Anfrage stoppt nicht den gesamten Update-Prozess.

==== Fehlerbehandlungsstrategie pro Ebene

[cols="2,3,4", options="header"]
|===
| Ebene | Fehlertyp | Verhalten

| *Team-Ebene* (cronUpdate.js)
| Fehler bei der Verarbeitung eines Teams (beliebige Exception)
| `console.error` mit Stack-Trace; das Team wird übersprungen; alle anderen Teams werden normal verarbeitet. Implementiert via `try/catch` im `for...of`-Loop.

| *API-Ebene* (apiClient.js)
| HTTP-Fehler (4xx, 5xx), Netzwerkfehler (ECONNRESET, ETIMEDOUT)
| `fetchTeamMatches` → `[]`; `fetchMatchInfo` → `null`; `fetchClubTeams` → `null`. Fehlerdetails werden via `console.error` geloggt (HTTP-Status oder Fehlermeldung).

| *Cache-Ebene* (storage.js)
| Abgelaufener Cache, kein Cache, defekte Cache-Datei (JSON-Parse-Fehler)
| Abgelaufener Cache: wird als Fallback zurückgegeben mit `stale: true`. Kein Cache: `{ teams: null, stale: false }`. Parse-Fehler: `{ teams: null, stale: true }` — impliziter API-Aufruf.

| *Club-Teams-API-Fehler* (cronUpdate.js)
| `fetchClubTeams` gibt `null` zurück, aber abgelaufener Cache vorhanden
| `[WARN]` Meldung; abgelaufene Cache-Daten werden als Fallback verwendet — letzter bekannter Zustand bleibt aktiv.

| *Club-Teams-API-Fehler* (cronUpdate.js)
| `fetchClubTeams` gibt `null` zurück, kein Cache vorhanden
| `[ERROR]` Meldung; leeres Array — kein ICS-Update in diesem Lauf. Bestehende ICS-Dateien bleiben unverändert.

| *ICS-Generierung* (icsGenerator.js)
| `matches` ist leer oder `null`
| `generateICS` gibt `null` zurück; `cronUpdate.js` prüft auf `null` vor `saveICS` und loggt `[WARN]`.
|===

==== Konsequenz für bestehende ICS-Abonnenten

Bei einem komplett fehlgeschlagenen Update-Lauf (z. B. API dauerhaft nicht erreichbar) werden bestehende ICS-Dateien auf GitHub Pages *nicht überschrieben*. Abonnenten sehen weiterhin die zuletzt erfolgreich generierten Daten. Erst bei einem erfolgreichen nächsten Lauf (alle 6 Stunden) werden die Dateien aktualisiert.

=== Caching-Strategie (Zwei-Layer-Cache)

Das System verwendet konzeptionell zwei Cache-Schichten:

==== Layer 1: Team-Metadaten-Cache (30-Tage-TTL)

* *Datei:* `generated/teams-cache.json`
* *Inhalt:* `{ cachedAt: ISO-8601-String, teams: [{ id, name, ageGroup }, ...] }`
* *TTL:* 30 Tage (2.592.000.000 Millisekunden)
* *Invalidierung:* Ausschließlich zeitbasiert (TTL-Ablauf)
* *Fallback:* Abgelaufener Cache wird bei API-Fehler weiterverwendet
* *Wirksamkeit:* In GitHub Actions nicht wirksam (ephemerer Runner); primär für lokale Entwicklung

==== Layer 2: ICS-Dateien auf GitHub Pages (kein explizites Expiry)

* *Dateien:* `generated/{teamId}_{type}.ics`
* *Lebensdauer:* Bis zum nächsten erfolgreichen Update-Lauf
* *Cache-Control:* Wird von GitHub Pages gesetzt (typischerweise kurze TTL für statische Assets)
* *Funktion:* Abonnenten erhalten bei temporärer GitHub-Pages-Nichterreichbarkeit die zuletzt gecachten Daten durch ihren Kalender-Client

=== Sicherheit

==== Input-Validierung in storage.js

Alle externen Eingaben, die als Teil von Dateipfaden verwendet werden, werden validiert:

[source,javascript]
----
const VALID_TYPES = new Set(['all', 'home', 'away']);

function saveICS(teamId, type, data) {
  if (!VALID_TYPES.has(type)) throw new Error(`Ungültiger ICS-Typ: ${type}`);
  if (!/^\d+$/.test(String(teamId))) throw new Error(`Ungültige teamId: ${teamId}`);
  const filepath = path.join(ICS_DIR, `${teamId}_${type}.ics`);
  // ...
}
----

* `type` wird gegen eine Whitelist geprüft (`Set`-Lookup in O(1)) — verhindert beliebige Pfadsegmente
* `teamId` wird gegen `/^\d+$/` geprüft — erlaubt ausschließlich Ziffern, verhindert `../` und andere Path-Traversal-Muster

==== HTML-Escaping in generateHTML.js

Alle aus der API stammenden Strings, die in HTML-Output eingebettet werden, durchlaufen `escapeHtml()`:

[source,javascript]
----
function escapeHtml(str) {
  return String(str)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
----

* Verhindert XSS-Angriffe in der generierten HTML-Seite
* Betrifft: Teamnamen, Altersgruppen-Bezeichnungen, Dateinamen in Link-Attributen

==== ICS-Escaping in icsGenerator.js

Alle Freitext-Felder in ICS-Properties werden durch `icsEscape()` normiert:

[source,javascript]
----
function icsEscape(text) {
  if (!text) return '';
  return text
    .replace(/\\/g, '\\\\')  // Backslash → \\
    .replace(/;/g, '\\;')    // Semikolon → \;
    .replace(/,/g, '\\,')    // Komma → \,
    .replace(/\n/g, '\\n')   // Newline → \n (literal)
    .replace(/\r/g, '');     // Carriage Return entfernen
}
----

* Verhindert, dass Sonderzeichen die ICS-Struktur brechen (Property-Injection)
* Betrifft: Kalendernamen (X-WR-CALNAME), HTML-Beschreibungen (X-ALT-DESC)
* Für DESCRIPTION übernimmt die `ics`-Bibliothek das Escaping

==== Kein Secret-Management erforderlich

Die Basketball-Bund-API erfordert keine Authentifizierung. Es werden keine API-Keys, Tokens oder Passwörter verwendet. Das Repository enthält keine sensiblen Daten — lediglich die öffentlich bekannte `clubId: 4468`.

| Level | Verwendung | Beispiel

| `console.log` / `[DEBUG]`

* 8 strukturelle Einzelprüfungen (Seitentitel, `lang`-Attribut, Landmarks, `<h1>`, `aria-controls`-Integrität, Tablist-Label, Tastaturnavigation, `alt`-Attribute)

=== Zeitzonenbehandlung

Alle Spieltermine werden in der deutschen Lokalzeit (Europe/Berlin) gespeichert. Die ICS-Dateien verwenden `DTSTART` und `DTEND` ohne UTC-Konvertierung, kombiniert mit:

* `startInputType: 'local'` und `startOutputType: 'local'` in der ics-Bibliothek
* `X-WR-TIMEZONE:Europe/Berlin` im VCALENDAR-Header

Dies bedeutet, dass die generierten Zeiten "floating" sind — sie werden von Kalender-Clients in der lokalen Zeitzone des Abonnenten interpretiert, nicht als UTC. Für Abonnenten in Deutschland (Europe/Berlin) ist das korrekt. Abonnenten in anderen Zeitzonen würden falsche Zeiten angezeigt bekommen — dies ist ein bekanntes Trade-off zugunsten der Einfachheit.

== Architekturentscheidungen

=== ADR-001: ICS (RFC 5545) statt CalDAV als Ausgabeformat

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2024

| *Kontext*
| Spieltermine der Fibalon Baskets Neumarkt sollen für Fans und Vereinsmitglieder in deren persönliche Kalender-Apps abrufbar sein. Dabei sollen verschiedene Plattformen (iOS, Android, Windows, macOS) unterstützt werden, ohne dass Nutzer eine App installieren müssen.

| *Entscheidung*
| Verwendung von statischen ICS-Dateien nach RFC 5545 als abonnierbare Kalender-Feeds. Keine CalDAV-Server-Implementation.

| *Begründung*
a|
* ICS wird nativ von iOS Calendar, Google Calendar, Outlook, Thunderbird und praktisch allen anderen Kalenderanwendungen unterstützt.
* CalDAV erfordert einen dauerhaft laufenden Server mit Authentifizierung, Session-Management und Datenbank.
* Statische ICS-Dateien können auf GitHub Pages gehostet werden — kostenlos und wartungsfrei.
* ICS ist ein Pull-Protokoll: der Client fragt die URL periodisch ab. Server-seitige Push-Logik ist nicht erforderlich.
* Ein Ausfall des Generators hat keine sofortige Auswirkung auf Abonnenten (bestehende Daten bleiben abrufbar).

| *Konsequenzen*
a|
* Positive: Keine Infrastrukturkosten, universelle Kompatibilität, kein laufender Server.
* Negative: Keine Echtzeit-Updates (Client bestimmt die Sync-Frequenz, typischerweise stündlich). Keine bidirektionale Synchronisation möglich (Nutzer können keine Events bearbeiten).
|===

=== ADR-002: Eine clubId als einzige Konfiguration

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2024

| *Kontext*
| Das System soll für einen bestimmten Verein (Fibalon Baskets Neumarkt, clubId: 4468) betrieben werden. Es muss entschieden werden, wie Teams konfiguriert werden.

| *Entscheidung*
| Die gesamte Konfiguration besteht aus `config.json` mit einer einzigen Variable: `{ "clubId": "4468" }`. Teams werden dynamisch über die Club-API ermittelt.

| *Begründung*
a|
* Die Basketball-Bund-API bietet einen Endpunkt, der alle Teams eines Vereins liefert — manuelle Pflege ist nicht nötig.
* Neue Teams (z. B. neue Altersklassen) werden automatisch beim nächsten Cache-Ablauf erkannt.
* Einfachheit hat Priorität: ein Parameter statt einer gepflegten Teamliste.
* Die `clubId` ist eine stabile, unveränderliche Kennung des Vereins im DBB-System.

| *Konsequenzen*
a|
* Positive: Zero-Maintenance bei Teamänderungen, eine einzige Konfigurationsdatei, einfaches Onboarding für neue Vereine.
* Negative: Bei strukturellen Änderungen der Club-API müssen die Felder neu gemappt werden. Teams, die in den nächsten 150 Tagen keine Spiele haben, erscheinen nicht im Feed.

=== ADR-003: 30-Tage-Cache für Team-Metadaten

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2024

| *Kontext*
| Die Club-Teams werden bei jedem der täglich vier Update-Zyklen benötigt. Die Team-Zusammensetzung eines Vereins ändert sich selten (typischerweise einmal pro Saison).

| *Entscheidung*
| Team-Metadaten werden in `generated/teams-cache.json` mit einer TTL von 30 Tagen zwischengespeichert.

| *Begründung*
a|
* Ohne Cache: ~120 zusätzliche API-Aufrufe pro Monat an `/actualmatches` nur für Team-Ermittlung.

* Bei API-Fehler kann der abgelaufene Cache als Notfall-Fallback dienen.

| *Konsequenzen*
a|
* Positive: Reduzierte API-Last, Ausfalltoleranz bei temporären API-Problemen, schnellere Ausführung.
* Negative: In GitHub Actions ist der Cache wirkungslos (ephemerer Runner). Wenn ein Team hinzukommt, erscheint es erst nach Cache-Ablauf (maximal 30 Tage verzögert). Dies ist als technische Schuld dokumentiert.

=== ADR-004: Differenzierte Alarm-Zeiten (Heim 30 min / Auswärts 60 min)

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2024

| *Kontext*
| Kalender-Events sollen einen Alarm (VALARM) enthalten, der Abonnenten rechtzeitig an das Spiel erinnert. Bei Heimspielen ist die Sporthalle in der Nähe; bei Auswärtsspielen muss mehr Anreisezeit eingeplant werden.

| *Entscheidung*
| VALARM-Trigger: 30 Minuten vor `DTSTART` bei Heimspielen, 60 Minuten bei Auswärtsspielen. `DTSTART` liegt bereits 60 Minuten vor dem eigentlichen Kickoff.

| *Begründung*
a|
* Heimspielfans/-besucher sind oft in der Nähe der Halle — 30 min Vorlauf zur Erinnerung ausreichend.
* Auswärtsspiele erfordern Anreise — 60 min Vorlauf gibt mehr Puffer.
* Die Unterscheidung Heim/Auswärts ist bereits per `teamPermanentId`-Vergleich bekannt.
* Da `DTSTART` 60 min vor Kickoff liegt, entspricht ein Alarm 30 min vor `DTSTART` dem tatsächlichen Zeitraum von 90 min vor Kickoff bei Heimspielen, bzw. 120 min vor Kickoff bei Auswärtsspielen.

| *Konsequenzen*
a|
* Positive: Individuell sinnvolle Erinnerungszeiten je nach Spieltyp.
* Negative: Nur ein Alarm pro Event (keine mehrstufigen Erinnerungen). Abonnenten können den Alarm in ihrer Kalender-App manuell anpassen.
|===

=== ADR-005: Event-Zeitfenster (DTSTART = Kickoff −1h, DTEND = Kickoff +2,5h)

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2024

| *Kontext*
| Ein Basketballspiel dauert offiziell 40 Minuten (4 × 10 Min.) plus Pausen. Der Termin im Kalender soll den realen Aufwand eines Spielbesuchs abbilden: Anreise + Anpfiff + Spiel + Nachspielzeit.

| *Entscheidung*
| `DTSTART` = Kickoff − 60 Minuten (Anreise-/Aufwärmpuffer), `DTEND` = Kickoff + 150 Minuten (2,5 Stunden für Spielzeit inkl. Pausen und Nachspielzeit).

| *Begründung*
a|
* 60 Minuten Vorlauf: Typische Anreisezeit plus Einlass plus Aufwärmen der Teams.
* 150 Minuten Gesamtspieldauer: 40 Min. Spielzeit + 15 Min. Halbzeit + Verlängerungen + Auslaufen = ca. 90–120 Min. aktive Spielzeit, plus Puffer.
* Dieses Zeitfenster blockiert den Kalender realistisch und vermeidet Überschneidungen mit anderen Terminen.

| *Konsequenzen*
a|
* Positive: Realistisches Zeitfenster im Kalender, sinnvolle Alarm-Berechnung.
* Negative: Das Zeitfenster ist für alle Spiele gleich — Überstunden (Verlängerung, Overtime) werden nicht dynamisch erfasst. Die Kickoff-Zeit stammt aus der API; Zeitplanänderungen kurzfristig vor dem Spiel werden erst beim nächsten Update-Zyklus reflektiert.
|===

=== ADR-008: Logo-URL-Konstruktion aus teamPermanentId statt API-Aufruf

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2026-04-11

| *Kontext*
| Die Übersichtsseite sollte Vereins- und Team-Logos anzeigen. Ursprünglich war `fetchClubInfo()` vorgesehen, das den Club-Endpunkt abfragen sollte. Der Endpunkt (`/rest/club/id/{clubId}/actualmatches`) liefert jedoch kein Logo-URL-Feld.

| *Entscheidung*
| Logos werden nicht per API-Call abgerufen, sondern die URL direkt aus der bereits bekannten `teamPermanentId` konstruiert: `https://www.basketball-bund.net/media/team/{teamPermanentId}/logo`. `fetchClubInfo()` wurde vollständig entfernt.

| *Begründung*
a|
* Das Basketball-Bund-Medien-Endpoint liefert unter dieser URL zuverlässig das Logo.
* Alle Teams eines Vereins teilen dasselbe Club-Logo — jede `teamPermanentId` desselben Vereins zeigt auf dasselbe Bild.
* Kein zusätzlicher API-Call erforderlich: die `teamPermanentId` ist bereits durch `fetchClubTeams()` bekannt.
* Einfacher und ausfallsicherer: kein Logo-Fetch kann fehlschlagen (Bild wird lazy durch den Browser geladen).

| *Konsequenzen*
a|
* Positive: Kein zusätzlicher API-Aufruf, keine zusätzliche Fehlerbehandlung, `apiClient.js` schlanker.
* Negative: Wenn basketball-bund.net die Media-URL-Struktur ändert, müssen alle Logo-URLs angepasst werden. Das Logo-Bild wird vom Browser beim Öffnen der Seite geladen (kann scheitern, falls URL nicht erreichbar — `<img>` zeigt dann keinen Broken-Image-Indikator da `alt=""`).

=== ADR-009: Spielplan-Anzeige in Team-Cards (Saison-gefiltertes matches-Array)

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Akzeptiert

| *Datum*
| 2026-04-11

| *Kontext*
| Die Übersichtsseite soll pro Team den Spielplan der aktuellen Saison anzeigen — vergangene Spiele mit Ergebnis, kommende ohne — direkt in den Tab-Panels, oberhalb der Kalender-Abo-Buttons.

| *Entscheidung*
| `cronUpdate.js` filtert die rohen API-Matches auf `seasonId === 2025`, sortiert sie aufsteigend und speichert sie als `matches[]`-Array in `metadata.json`. `generateHTML.js` rendert dieses Array als HTML-Tabelle im jeweiligen Tab-Panel (Alle/Heim/Auswärts). Liga-Erkennung per Namens-Heuristik: enthält `liganame` das Wort "Liga" → Liga-Typ, sonst Cup/Pokal. Cup-Farbe ist konfigurierbar via `config.theme.cupColor` (Default `#7c3aed`).

| *Begründung*
a|
* Matches sind bereits per `fetchTeamMatches()` geladen — kein zusätzlicher API-Aufruf nötig.
* Saison-Filter (`seasonId === 2025`) ist im Daten-Layer (`cronUpdate.js`), nicht im View-Layer — `generateHTML.js` bekommt nur saubere, vorverarbeitete Daten.
* `isNext`-Flag kennzeichnet das nächste anstehende Spiel bereits beim Schreiben der Metadaten — keine Zeitberechnungen im Template nötig.
* Konfigurierbare Cup-Farbe (`config.theme.cupColor`) ermöglicht Anpassung ohne Code-Änderung.
* Harte Saison-Konstante (`CURRENT_SEASON = 2025`) auf Modul-Ebene ist dokumentierter Trade-off: einfach, aber erfordert jährliche Code-Änderung.

| *Konsequenzen*
a|
* Positive: Spielplan ohne zusätzliche API-Calls, Tab-Filterung (Alle/Heim/Auswärts) konsistent mit Kalender-Abo-Struktur, WCAG-konform (axe-core-geprüft).
* Negative: `CURRENT_SEASON` muss jährlich aktualisiert werden. `metadata.json` wächst durch das `matches`-Array (bei typisch 14–20 Spielen pro Team vernachlässigbar). Liga-Heuristik ("liga" im Namen) kann bei ungewöhnlichen Wettbewerbsnamen fehlschlagen.
|===

=== ADR-006: X-ALT-DESC für HTML-Beschreibungen (aktuell deaktiviert)

[cols="2,5", options="header"]
|===
| Aspekt | Inhalt

| *Status*
| Deferred (vorläufig deaktiviert, Code vorhanden)

| *Datum*
| 2024

| *Kontext*
| ICS `DESCRIPTION` unterstützt nur Plain-Text. Microsoft Outlook und einige andere Clients unterstützen die nicht-standardisierte X-Property `X-ALT-DESC;FMTTYPE=text/html`, die eine HTML-Version der Beschreibung enthält und eine reichere Darstellung ermöglicht.

| *Entscheidung*
| Die Infrastruktur für `X-ALT-DESC` ist vollständig implementiert (`createHtmlDescription()`, Post-Processing-Logik in `generateICS()`), aber in der Event-Erzeugung temporär auskommentiert (`// htmlDescription`).

| *Begründung*
a|
* `X-ALT-DESC` ist keine RFC-5545-Property — manche Clients ignorieren sie, andere verarbeiten sie fehlerhaft.
* Debugging-Schwierigkeiten: HTML in ICS-Strings ist schwer zu verifizieren.
* Implementierung ist fertig, Aktivierung erfordert Verifikation auf allen Zielclients.
* Die Deaktivierung ist ein bewusstes temporäres Trade-off — Plain-Text DESCRIPTION ist universell kompatibel.

| *Konsequenzen*
a|
* Aktuelle Konsequenz: Alle Clients zeigen Plain-Text-Beschreibungen. Funktionalität vollständig erhalten.
* Zukünftige Aktivierung: Muss `buildEvent()` `htmlDescription` wieder befüllen und `createHtmlDescription()` testen. Besondere Sorgfalt bei Line-Folding (75-Zeichen-Limit) und HTML-in-ICS-Escaping erforderlich.
|===

| ICS-Dateien verwenden floating times mit `X-WR-TIMEZONE:Europe/Berlin`. Korrekte Interpretation liegt beim Client. Eigentliche IANA-Zeitzonensemantik (VTIMEZONE-Komponente) ist nicht implementiert.

== Glossar

[cols="2,5", options="header"]
|===
| Begriff | Erklärung

| *RFC 5545*
| "Internet Calendaring and Scheduling Core Object Specification". Der Internet-Standard (Request for Comments Nr. 5545 der IETF, veröffentlicht 2009), der das iCalendar-Datenformat definiert. Legt Syntax, Pflichtfelder und Escape-Regeln für Kalender-Dateien fest. Ablösung von RFC 2445. Wichtigste Einschränkung: Textzeilen dürfen maximal 75 Oktetts lang sein (Line-Folding erforderlich).

| *ICS*
| Dateiendung und umgangssprachliche Bezeichnung für iCalendar-Dateien nach RFC 5545. Der MIME-Type ist `text/calendar`. ICS-Dateien können einzelne oder mehrere Kalender-Ereignisse (VEVENT) enthalten und von allen gängigen Kalenderanwendungen importiert oder abonniert werden.

| *VCALENDAR*
| Das äußere Container-Objekt einer ICS-Datei, begrenzt durch `BEGIN:VCALENDAR` und `END:VCALENDAR`. Enthält Calendar-Properties wie `PRODID`, `VERSION`, `CALSCALE`, `METHOD` und X-Properties wie `X-WR-CALNAME`, sowie alle enthaltenen VEVENT-Objekte.

| *VEVENT*
| Ein einzelnes Kalenderereignis innerhalb einer ICS-Datei, begrenzt durch `BEGIN:VEVENT` und `END:VEVENT`. Pflichtfelder nach RFC 5545: `DTSTART` (Startzeit), `UID` (eindeutige ID), `DTSTAMP` (Zeitstempel der Erzeugung). Im BBB ICS Generator enthält jedes VEVENT auch `DTEND`, `SUMMARY`, `DESCRIPTION`, `LOCATION`, `BUSYSTATUS` und ein `VALARM`.

| *VALARM*
| Eine Unterkomponente eines VEVENT, die eine Erinnerung (Alarm) definiert, begrenzt durch `BEGIN:VALARM` und `END:VALARM`. Felder: `ACTION` (z. B. `DISPLAY`), `DESCRIPTION` (Alarmtext), `TRIGGER` (relativer Zeitversatz, z. B. `-PT30M` für 30 Minuten vor DTSTART).

| *UID*
| "Unique Identifier". Pflichtfeld in VEVENT nach RFC 5545. Muss global eindeutig sein und über alle Versionen eines Events konstant bleiben, damit Kalender-Clients Updates (nicht Duplikate) erkennen. Im BBB ICS Generator: `{matchId}@basketball-bund.net`.

| *webcal://*
| Ein URI-Schema (kein eigenes Protokoll), das HTTPS für Kalender-Subscriptions signalisiert. `webcal://` ist funktional identisch mit `https://`, wird aber von iOS Calendar, macOS Calendar und Outlook als Trigger für die Subscription-Logik erkannt. Der Browser öffnet beim Klick automatisch die Kalender-App.

| *teamPermanentId*
| Die dauerhafte, unveränderliche ID eines Teams im Basketball-Bund-System. Im Gegensatz zu `teamId` bleibt die `teamPermanentId` über Saisons hinweg stabil. Im BBB ICS Generator als `teamId` verwendet und als Dateinamen-Bestandteil (`{teamPermanentId}_{type}.ics`).

| *clubId*
| Die eindeutige ID eines Basketballvereins im DBB-System. Im BBB ICS Generator die einzige Konfigurationsvariable (`config.json`). Für die Fibalon Baskets Neumarkt: `4468`. Alle Teams, ICS-Feeds und Links werden von dieser einen ID abgeleitet.

| *spielfeld*
| Feldname im Basketball-Bund API-Response (`matchInfo.matchInfo.spielfeld`). Enthält Informationen zur Spielhalle: `bezeichnung` (Name der Halle), `strasse` (Straße), `plz` (Postleitzahl), `ort` (Stadt). Wird in der `LOCATION`-Property des VEVENT und in der DESCRIPTION verwendet.

| *Line-Folding*
| Pflicht-Mechanismus nach RFC 5545: ICS-Textzeilen dürfen maximal 75 Oktetts lang sein. Längere Zeilen müssen mit einem CRLF gefolgt von einem einzelnen Leerzeichen oder Tab umgebrochen werden. Die `ics`-Bibliothek übernimmt Line-Folding für Standard-Properties. Bei manuell eingefügten X-Properties (z. B. `X-ALT-DESC`) muss Line-Folding manuell berücksichtigt werden.

| *X-Property*
| Eine proprietäre, nicht-standardisierte Property in einer ICS-Datei, erkennbar am `X-`-Präfix. Erlaubt durch RFC 5545 als Erweiterungsmechanismus. Beispiele im BBB ICS Generator: `X-WR-CALNAME` (Kalenderanzeigename), `X-WR-TIMEZONE` (bevorzugte Zeitzone), `X-WR-CALDESC` (Kalenderbeschreibung). Werden von Kalender-Clients unterschiedlich stark unterstützt.

| *X-ALT-DESC*
| Eine spezifische X-Property mit dem Wert `FMTTYPE=text/html`, die eine HTML-Version der DESCRIPTION enthält. Wird von Microsoft Outlook und einigen anderen Clients für eine reichere Darstellung verwendet. Nicht im RFC 5545 definiert. Im BBB ICS Generator implementiert aber vorläufig deaktiviert (ADR-006).

| *ICS-Escape*
| Die Zeichenmaskierung, die innerhalb von ICS-Property-Werten erforderlich ist, um Syntaxkonflikte zu vermeiden. Nach RFC 5545 müssen folgende Zeichen in Textwerten escapt werden: Backslash (`\`) → `\\`, Semikolon (`;`) → `\;`, Komma (`,`) → `\,`, Newline (`\n`) → `\n` (literal zwei Zeichen). Implementiert in `icsGenerator.js::icsEscape()`.

| *PRODID*
| "Product Identifier". Pflichtfeld im VCALENDAR-Container nach RFC 5545. Identifiziert das Programm, das die ICS-Datei erzeugt hat. Im BBB ICS Generator: `-//bbb-ics-generator//DE`. Format: `-//Organisation//Produkt//Sprache`.

| *CALSCALE*
| "Calendar Scale". VCALENDAR-Property, die den verwendeten Kalender-Standard angibt. Standardwert: `GREGORIAN` (Gregorianischer Kalender). Im BBB ICS Generator explizit auf `GREGORIAN` gesetzt.

| *METHOD*
| VCALENDAR-Property, die den Verwendungszweck der ICS-Datei beschreibt. Wert `PUBLISH` bedeutet: der Kalender wird veröffentlicht (kein iTIP-Workflow). Alternative: `REQUEST` für Meeting-Einladungen. Im BBB ICS Generator: `METHOD:PUBLISH`.

| *DTSTART / DTEND*
| "Date-Time Start / End". Pflicht- bzw. Optionalfelder im VEVENT. Definieren Beginn und Ende eines Kalendereintrags. Können als lokale Zeit (floating), UTC-Zeit (mit `Z`-Suffix) oder mit expliziter Zeitzone (`TZID`-Parameter) angegeben werden. Im BBB ICS Generator: floating times mit `startInputType/startOutputType: 'local'`.

| *DTSTAMP*
| "Date-Time Stamp". Pflichtfeld im VEVENT nach RFC 5545. Enthält den UTC-Zeitpunkt, zu dem das Event-Objekt erzeugt wurde. Wird von der `ics`-Bibliothek automatisch gesetzt.

| *Kickoff*
| Fachbegriff für den Anpfiff-Zeitpunkt eines Basketballspiels, wie er aus der Basketball-Bund-API als `kickoffDate` (Datum) und `kickoffTime` (Uhrzeit, Format `HH:MM`) geliefert wird. `DTSTART` im generierten Event liegt 60 Minuten vor dem Kickoff.

| *teamname / teamnameSmall*
| Felder im Basketball-Bund API-Response. `teamname` ist der vollständige Teamname (z. B. "Fibalon Baskets Neumarkt"). `teamnameSmall` ist die Kurzform (z. B. "Baskets Neumarkt"), die im VEVENT-`SUMMARY` verwendet wird. Wenn `teamnameSmall` nicht verfügbar ist, wird `teamname` als Fallback verwendet.
|===
